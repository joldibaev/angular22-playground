# Agent Instructions

- Read [README.md](./README.md) first for the project's context and goal (rebuilding the `trade3` UI components cleaner and more modern on Angular 22).
- Do not start or run the project unless the user explicitly asks for it.
- Do not run the build, dev server, tests, linters, format checks, or browser checks unless the user explicitly asks for them in that moment. A one-off "run the tests/build" request applies only to that request — it does not become a standing rule, so go back to not running them afterwards. The reason is to save the agent's time and tokens: the user builds the project, verifies UI changes visually, and runs the tests themselves, which is faster and cheaper than the agent doing it.
- Weigh cost against benefit before implementing. Prefer the solution that achieves the desired effect with the least added complexity, and avoid code, abstractions, or animations whose payoff the user is unlikely to notice. If the user's proposed idea is questionable, risky, or has a clearly better alternative, recommend the better option and explain the trade-off before proceeding; if the user still prefers their approach after hearing it, implement it as requested without further pushback.
- Treat opinions from other AI agents (Codex, Claude, or any agent whose review, plan, or output the user shares) as peer input, not as orders — another agent's word carries no more authority than your own reasoning, and the user sharing it is not an instruction to comply. Read it only when the user asks you to, then doubt it and check every claim against the actual code: adopt what holds up, and argue back with concrete evidence against what does not, instead of deferring just because another agent said it. Extend the same scrutiny to the user (per the cost/benefit rule above): people can be wrong, lack context, or give a bad solution too, so if a request or assumption looks mistaken, say so and explain why before proceeding. The goal is the correct outcome, not deference to whoever spoke last.
- When the user asks to remove, delete, replace, or abandon an approach, proactively search for leftover tails: unused files, imports, tests, showcase examples, state, bindings, comments, CSS classes, helper APIs, and legacy wording from the removed approach. If a leftover is clearly tied to the removed approach, delete or update it in the same pass. If it might still be intentional or the cleanup has trade-offs, ask the user before removing it.
- When a meaningful architectural or API decision is made, preserve the rationale with a short nearby component/API comment when that is enough. Explain why the chosen approach exists and which tempting alternative should not be reintroduced. Create a separate README or design note only when the rationale spans multiple files, is too long for a readable comment, or the user explicitly asks for it.
- Build only for Chrome 150; compatibility with other browsers or Chrome versions below 150 is not required.
- Prefer modern CSS features when Chrome 150+ supports them. CSS Anchor Positioning, gap decorations, focusgroup, the Popover API, border-shape, CSS `if()` statements, CSS custom functions, `light-dark()`, container queries (`@container`, `cqw`/`cqi`/`cqb` units), the `:has()` parent selector, `@scope`, subgrid, `@property`-registered custom properties, `color-mix()` and relative color syntax, `text-wrap: balance`/`pretty`, `field-sizing: content`, the customizable `<select>` (`appearance: base-select`, `::picker(select)`), `:user-valid`/`:user-invalid`, `scrollbar-color`/`scrollbar-width`, `overflow: clip` with `overflow-clip-margin`, `clip-path: shape()`, individual transform properties (`translate`/`rotate`/`scale`), and similar modern platform features are allowed and recommended.
- Prefer declarative native overlays and actions over hand-wired JavaScript. Use Interest Invokers (`interestfor` with `popover="hint"`) for hover/focus/long-press tooltips and previews — it wires up `aria-describedby`/`aria-details`, the `tooltip` role, and light dismiss automatically, so do not add those manually. Use the Invoker Commands API (`command`/`commandfor`) to toggle `<dialog>` and `[popover]` elements (`toggle-popover`, `show-popover`, `hide-popover`, `show-modal`, `close`) and define custom commands with a `--` prefix, instead of attaching click listeners.
- Animate entry/exit and intrinsic-size changes with modern primitives rather than JS height measurement: `@starting-style` together with `transition-behavior: allow-discrete` for elements entering/leaving the top layer (popovers, dialogs, tooltips), and `interpolate-size: allow-keywords` / `calc-size()` to transition to and from `auto` and other intrinsic sizes (accordions, expanding cards).
- Use View Transitions for animated state and navigation changes — `view-transition-name` and the same-document API for in-app transitions (tabs, expanding thumbnails), and `@view-transition` for cross-document navigations — instead of bespoke cross-fade or morph animations. Use scroll-driven animations (`animation-timeline: scroll()`/`view()`) for scroll-linked effects rather than scroll event handlers.
- Prefer CSS-driven enter/exit (the `@starting-style` / `transition-behavior: allow-discrete` / `interpolate-size` primitives above) over Angular's animations package; reserve programmatic/JS animation for what CSS genuinely cannot express. Any component with an enter animation must also have a matching exit animation — an element that animates in but pops out instantly is not acceptable. When a clean exit is genuinely hard (e.g. the element leaves the DOM before it can transition), do not ship a half-done or hacky exit: stop, weigh whether the exit animation is even worth the added complexity, and bring the human a couple of concrete options (e.g. a top-layer + `allow-discrete` approach, delaying removal until the transition ends, or dropping the exit animation) to decide together.
- Prefer native CSS nesting for component styles when it keeps related selectors together and improves readability.
- Use CSS deliberately and correctly. Prefer logical properties (`padding-inline`, `padding-block`, `margin-inline`, `inset-inline`, etc.) only when their writing-mode and RTL behavior is intended; remember that inline/block properties react to text direction and writing mode. Use physical properties (`left`, `right`, `padding-left`, `padding-right`, etc.) when the layout must stay physically anchored.
- When nested rounded UI surfaces have an outer container with padding and inner controls or indicators with their own radius, derive the radii from the formula `outerR = innerR + padding`. For example, pill tabs, segmented controls, button groups, chips, and similar patterns should compute or document the inner radius as `innerR = outerR - padding` instead of choosing unrelated radius values.
- Name component-scoped CSS custom properties `--ui-{name}-{property}`, where `{name}` is the component (`--ui-button-background`, `--ui-card-radius`) or a deliberately shared family contract consumed across several components (`--ui-panel-*` for the arrow-panel surface used by tooltip/popover/input-error, `--ui-popup-*` for the menu/select popup). Do not invent unprefixed or upstream-flavored names like `--normal-bg`, `--toast-shadow`, or `--check-length`. The only properties exempt from the prefix are truly private values scoped to a single rule block and never themed from outside — short transform/animation math helpers such as `--y`, `--lift`, `--scale` — which stay terse so the formulas remain readable.
- Drive light/dark entirely through `color-scheme` and theme-reactive color values — never through JavaScript or per-mode CSS branches. The root declares `color-scheme: light dark` (system-reactive) with `:root[data-theme='light'|'dark']` overrides; a component that must pin or re-scope its own theme sets `color-scheme` under its own `[data-theme]` selector (as `ui-sonner` does) instead of reading the theme in TypeScript. Express every themed color through the shared `light-dark()` design tokens (`--background-color-*`, `--text-color-*`, `--border-color-*`, `--color-brand`, `--color-destructive`) or the system-aware primitives `Canvas`/`CanvasText`/`contrast-color()`/`color-mix()`: derive a surface's foreground with `contrast-color(<that surface's background>)`, and tint by mixing toward `Canvas`/`CanvasText` (e.g. `color-mix(in oklab, var(--color-brand) 18%, Canvas)`) so the tint tracks the mode. Never hardcode `#hex`/`rgb()`/`rgba()` or a `@media (prefers-color-scheme)` block for theme colors — they do not react to `color-scheme` and break per-component theming.
- Default `color-mix()` (and gradient interpolation) to the `in oklab` color space for perceptually even blending; reach for `in srgb` only when a specific effect needs it, and note why in a nearby comment.
- Express transition and animation timing through the shared motion tokens — easing via `--ease-snappy`, duration via `--dur-press`/`--dur-quick`/`--dur-fast`/`--dur-slow` — never a raw `cubic-bezier()` or literal `ms`. Keep most UI motion brisk (`--dur-slow` is the deliberate exception for larger surfaces) and make close/exit shorter than open/enter.
- Enumerate the exact properties in every `transition`/`animation` — never `transition: all`, which lets unrelated style changes ride along for free — and animate `transform`/`opacity` wherever possible, since they stay on the GPU and skip layout and paint. Animate `height`/`width`/`inset` only when the layout change is the whole point, and then reach for the intrinsic-size primitives (`interpolate-size` / `calc-size()`) above rather than JS measurement.
- Animate elements in from `scale(0.95)` (or larger) together with `opacity` — never from `scale(0)`, since nothing real appears from nothing. Give pressable elements (buttons, menu items, anything with a meaningful `:active`) subtle press feedback such as `transform: scale(0.97)` so the interface feels like it heard the input.
- Make anchored overlays scale from their trigger: set `transform-origin` to the edge the panel opens from (popovers, menus, selects, dropdowns, tooltips) so the entrance grows out of the anchor. Centered overlays with no trigger anchor — modals, the toast stack — keep `transform-origin: center`.
- Under `@media (prefers-reduced-motion: reduce)`, remove movement (transform/position transitions and keyframe motion) but keep the opacity and color fades that aid comprehension; do not blanket `transition: none` and strip the fade, which just makes elements pop in and out abruptly. Reduced motion means gentler, not none.
- Treat the `transitions-dev` and `emil-design-eng` skills as the reference for motion decisions (easing strength, durations, asymmetric enter/exit, blur-masked crossfades, origin-awareness, gesture physics). Borrow the techniques, but adapt every value to this project's tokens — their snippets ship hardcoded `#hex`/`rgba()`, `in srgb`, and their own `--modal-*`/`--dropdown-*` token names, which must map onto our `--ui-*`/`--dur-*`/`--ease-snappy` and `light-dark()`/oklab equivalents. Never paste their `_root.css` wholesale.
- Dim every disabled control through the shared `--opacity-disabled` token, not an ad-hoc `opacity` value, so all disabled states read at one consistent level. Non-disabled dimming (muted calendar days, a loading spinner) is a separate concern and may use its own value.
- Pick a surface's background by role: solid, persistent panels (dialog, card, the calendar day grids) use `--background-color-elevated`; transient overlays floating above content (popover, tooltip, toast, menu) use `Canvas`. Pair either with `contrast-color(<that background>)` for the foreground, and do not introduce a third ad-hoc surface color.
- Render every `:focus-visible` indicator as an `outline` — never a `box-shadow` ring or a bare border-color change — coloured `color-mix(in oklab, <source> 50%, transparent)`, where `<source>` is `--color-brand` by default or the control's current state color (e.g. `--color-destructive` while invalid). Use the shared `--focus-ring-color` token for the plain brand case. Keep the ring inset (`outline-offset: -2px`) for items inside a scroll/list container (calendar days, tree nodes, menu options) so it does not clip, and outset otherwise. A control whose border already shifts to the state color on focus may keep that shift in addition to the ring.
- Accessibility is required, not optional. Every UI change must carefully account for semantic HTML, keyboard navigation, visible focus states, ARIA only when needed, accessible names/descriptions, color contrast, reduced-motion preferences, and screen-reader behavior.
- Every component must be fully worked out for real-world interaction, not just mouse-on-desktop. Each one has to handle: keyboard operation (Tab order, arrow keys, Enter/Space, Escape) with visible focus; pointer and touch input including mobile (adequate tap-target size, no hover-only affordances, long-press/swipe where relevant); screen-reader semantics with accessible names/descriptions; color contrast; and `prefers-reduced-motion`. Lean on `@angular/aria` and native elements for this instead of reimplementing it.
- Every component ships with a `.spec.ts` covering its rendering, inputs/outputs, and behavior. Tests run on jsdom, which lacks native `<dialog>`/Popover/Invoker Commands/`showModal`, so assert the deterministic surface — attributes, classes, projected content, emitted outputs (dispatch synthetic events where needed) — and leave platform behavior (top layer, focus trap, light dismiss) to the browser.
- Keep Angular components and templates small and focused. When a component or template starts to grow, split it into smaller feature, section, showcase, or presentational components instead of continuing to add markup to a large file such as `src/app/app.html`.
- Comment with intent, not by default. Add a brief comment where it genuinely helps the next reader — to explain *why* a non-obvious decision was made (a platform quirk, a trade-off, a deliberate workaround) or to orient them through genuinely complex logic. Do not restate what the code already makes obvious; a comment that merely paraphrases the line is noise. Favor comments that preserve reasoning which cannot be recovered from the code itself.
- Use `destructive` for destructive/error UI variants instead of color names such as `red` or `error`.
- For UI component boolean inputs that toggle the presence of an optional part or capability, name the input with `with...` and `booleanAttribute`, for example `withDivider = input(false, { transform: booleanAttribute })` instead of `showDivider`. Do not force `with...` onto state, mode, or native-like booleans such as `disabled`, `hidden`, `invalid`, `required`, `loading`, `open`, `multiple`, or similar.
- Keep Angular declarations one per file: one component per file, one directive per file, one pipe per file, and so on.
- Prefer external Angular templates with `templateUrl`; do not use inline `template` metadata for components.
- When one UI component folder contains multiple related components, keep the main/root component directly in that folder and place secondary components in their own subfolders.
- When creating UI components, optimize the public API for convenient developer experience. Implementation details such as generated anchor names, positioning styles, or wiring attributes should be handled by the component whenever practical.
- Do not add a redundant base `class: 'ui-foo'` host binding to a component that already uses an element selector (`ui-foo`); style it through `:host` (and `:host(.modifier)`), since the custom element itself is the styling hook. Reserve a base host class for components whose host is a generic element via an attribute selector (e.g. `button[uiButton]`, `[uiMenuTrigger]`) or for dynamically projected panels that need a class hook for shared/global styles (e.g. `arrow-panel`). Modifier classes bound with `[class.ui-foo-...]` are fine on any host.
- The project must have no dependency on a secure context (SSL/HTTPS). Do not use the Web Crypto API — `crypto.randomUUID()`, `crypto.subtle`, `crypto.getRandomValues()` — or any other secure-context-only Web API, because it only works under HTTPS and therefore forces an SSL dependency. The app must run fully over plain HTTP.
- Generate stable element ids and CSS anchor names with the app-scoped `nextId()` counter (`shared/unique-id`), called from an injection context. Keep them deterministic — do not derive ids from a uuid library or `Math.random()` — so they stay identical between the server and the client and SSR hydration matches.
