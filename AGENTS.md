# Agent Instructions

- Do not start or run the project unless the user explicitly asks for it.
- Do not run the build, dev server, tests, linters, format checks, or browser checks unless the user explicitly asks for them in that moment. A one-off "run the tests/build" request applies only to that request — it does not become a standing rule, so go back to not running them afterwards. The reason is to save the agent's time and tokens: the user builds the project, verifies UI changes visually, and runs the tests themselves, which is faster and cheaper than the agent doing it.
- Weigh cost against benefit before implementing. Prefer the solution that achieves the desired effect with the least added complexity, and avoid code, abstractions, or animations whose payoff the user is unlikely to notice. If the user's proposed idea is questionable, risky, or has a clearly better alternative, recommend the better option and explain the trade-off before proceeding; if the user still prefers their approach after hearing it, implement it as requested without further pushback.
- Build only for Chrome 149+; compatibility with other browsers or Chrome versions below 149 is not required.
- Prefer modern CSS features when Chrome 149+ supports them. CSS Anchor Positioning, gap decorations, focusgroup, the Popover API, border-shape, CSS `if()` statements, CSS custom functions, `light-dark()`, container queries (`@container`, `cqw`/`cqi`/`cqb` units), the `:has()` parent selector, `@scope`, subgrid, `@property`-registered custom properties, `color-mix()` and relative color syntax, `text-wrap: balance`/`pretty`, `field-sizing: content`, the customizable `<select>` (`appearance: base-select`, `::picker(select)`), `:user-valid`/`:user-invalid`, `scrollbar-color`/`scrollbar-width`, `overflow: clip` with `overflow-clip-margin`, `clip-path: shape()`, individual transform properties (`translate`/`rotate`/`scale`), and similar modern platform features are allowed and recommended.
- Prefer declarative native overlays and actions over hand-wired JavaScript. Use Interest Invokers (`interestfor` with `popover="hint"`) for hover/focus/long-press tooltips and previews — it wires up `aria-describedby`/`aria-details`, the `tooltip` role, and light dismiss automatically, so do not add those manually. Use the Invoker Commands API (`command`/`commandfor`) to toggle `<dialog>` and `[popover]` elements (`toggle-popover`, `show-popover`, `hide-popover`, `show-modal`, `close`) and define custom commands with a `--` prefix, instead of attaching click listeners.
- Animate entry/exit and intrinsic-size changes with modern primitives rather than JS height measurement: `@starting-style` together with `transition-behavior: allow-discrete` for elements entering/leaving the top layer (popovers, dialogs, tooltips), and `interpolate-size: allow-keywords` / `calc-size()` to transition to and from `auto` and other intrinsic sizes (accordions, expanding cards).
- Use View Transitions for animated state and navigation changes — `view-transition-name` and the same-document API for in-app transitions (tabs, expanding thumbnails), and `@view-transition` for cross-document navigations — instead of bespoke cross-fade or morph animations. Use scroll-driven animations (`animation-timeline: scroll()`/`view()`) for scroll-linked effects rather than scroll event handlers.
- Prefer native CSS nesting for component styles when it keeps related selectors together and improves readability.
- Use CSS deliberately and correctly. Prefer logical properties (`padding-inline`, `padding-block`, `margin-inline`, `inset-inline`, etc.) only when their writing-mode and RTL behavior is intended; remember that inline/block properties react to text direction and writing mode. Use physical properties (`left`, `right`, `padding-left`, `padding-right`, etc.) when the layout must stay physically anchored.
- When nested rounded UI surfaces have an outer container with padding and inner controls or indicators with their own radius, derive the radii from the formula `outerR = innerR + padding`. For example, pill tabs, segmented controls, button groups, chips, and similar patterns should compute or document the inner radius as `innerR = outerR - padding` instead of choosing unrelated radius values.
- Accessibility is required, not optional. Every UI change must carefully account for semantic HTML, keyboard navigation, visible focus states, ARIA only when needed, accessible names/descriptions, color contrast, reduced-motion preferences, and screen-reader behavior.
- Keep Angular components and templates small and focused. When a component or template starts to grow, split it into smaller feature, section, showcase, or presentational components instead of continuing to add markup to a large file such as `src/app/app.html`.
- Comment with intent, not by default. Add a brief comment where it genuinely helps the next reader — to explain *why* a non-obvious decision was made (a platform quirk, a trade-off, a deliberate workaround) or to orient them through genuinely complex logic. Do not restate what the code already makes obvious; a comment that merely paraphrases the line is noise. Favor comments that preserve reasoning which cannot be recovered from the code itself.
- Use `destructive` for destructive/error UI variants instead of color names such as `red` or `error`.
- For UI component boolean inputs that toggle the presence of an optional part or capability, name the input with `with...` and `booleanAttribute`, for example `withDivider = input(false, { transform: booleanAttribute })` instead of `showDivider`. Do not force `with...` onto state, mode, or native-like booleans such as `disabled`, `hidden`, `invalid`, `required`, `loading`, `open`, `multiple`, or similar.
- Keep Angular declarations one per file: one component per file, one directive per file, one pipe per file, and so on.
- Prefer external Angular templates with `templateUrl`; do not use inline `template` metadata for components.
- When one UI component folder contains multiple related components, keep the main/root component directly in that folder and place secondary components in their own subfolders.
- When creating UI components, optimize the public API for convenient developer experience. Implementation details such as generated anchor names, positioning styles, or wiring attributes should be handled by the component whenever practical.
- The project must have no dependency on a secure context (SSL/HTTPS). Do not use the Web Crypto API — `crypto.randomUUID()`, `crypto.subtle`, `crypto.getRandomValues()` — or any other secure-context-only Web API, because it only works under HTTPS and therefore forces an SSL dependency. The app must run fully over plain HTTP.
- Generate stable element ids and CSS anchor names with the app-scoped `nextId()` counter (`shared/unique-id`), called from an injection context. Keep them deterministic — do not derive ids from a uuid library or `Math.random()` — so they stay identical between the server and the client and SSR hydration matches.
